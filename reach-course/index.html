How does it know it's a cold email?

Reach reads the email body and scans for job-hunting keywords using a technique called fuzzy matching. If at least one job-related keyword group matches — words like “intern,” “recruit,” “apply,” or “hiring” — it considers the email cold outreach.

It also ignores emails to personal addresses like Gmail or Yahoo — those are probably to friends, not recruiters.

CODE 

export function isColdOutreach(body) {
  // One strong job-related keyword is enough signal
  return countKeywordMatches(body) >= 1;
}
PLAIN ENGLISH

This function answers a yes/no question: is this a cold outreach email?

It counts how many distinct job-related keyword groups appear in the email body.

If at least one keyword group matches — like finding "intern" or "hiring" — return true (yes, it's cold outreach).

💡
Key Insight: Heuristics over perfection

Reach doesn't use AI to classify emails — it uses a hand-crafted list of keywords with fuzzy matching. This makes it fast, free, and predictable. When steering AI on classification problems, ask yourself: does this need AI, or does a simple rule work well enough?

Meet the Characters

The five actors that power Reach — and exactly what each one is responsible for

Five specialists, one team

Reach is not one program — it's a team of five specialists. Each one has a specific job and only talks to the others when it needs to. Think of it like a newsroom: reporters, editors, printers, and distributors — each expert at one thing, working in sync.

📝
content.js — The Gmail Observer

Injected directly into Gmail. Watches for compose windows, detects sends, and injects the Reach sidebar widget. It's the only part of the app that can “see” what's on your screen.

⚡
background.js — The Service Worker

Runs silently in the background 24/7. Calls the Gmail API, runs the reply-check alarm every 30 minutes, and routes messages between the extension and the server.

🏗️
server/app.js — The Express Server

A Node.js API server that handles all the business logic. Saves outreach records, checks quotas, generates AI drafts, and manages user accounts.

🗄️
PostgreSQL Database — The Filing Cabinet

Stores everything permanently: users, outreach records, tracking pixels, and open events. Managed through Prisma, which translates JavaScript into database language.

🖥️
web/src — The React Dashboard

The kanban board you interact with at coldbase.live. Built with React — fetches data from the server and renders your outreach cards with drag-and-drop.

What gets saved — the data model

The database has four tables. Think of each one as a sheet in a spreadsheet — every row is one record, every column is a field.

👤

User

Email, password hash, Google ID, subscription plan (free/pro), Stripe IDs, quota usage. One row per person.

📨

Outreach

Company, contact name, email, subject, sent date, status (Sent / Replied / Interviewing / Offer / Ghosted), reply info, notes. One row per cold email sent.

🔍

TrackingPixel

Links each sent email to a unique tracking ID. When a recipient opens the email, their mail client loads an invisible image that records the event.

👁️

OpenEvent

Records every time the tracking pixel fires — timestamp, user agent, IP address. Many open events can belong to one TrackingPixel.

💡
Key Insight: Relationships between tables

One User has many Outreaches. One Outreach has one TrackingPixel. One TrackingPixel has many OpenEvents. This “one-to-many” relationship is the fundamental pattern in almost every database ever built.

03

How They Talk

The unusual communication patterns that connect Gmail, the extension, and your dashboard

The walkie-talkie problem

You'd think components in an app would just call each other directly — like picking up a phone and dialing. But browser extensions have a harder version of this problem. It's more like walkie-talkies: you can only send a message if the other person's radio is already on and listening.

Chrome extensions have three distinct environments that can't share memory directly:

1
The web page (Gmail)

content.js runs here. It can read the Gmail DOM but lives in a sandbox — it can't directly access Chrome APIs.

2
The extension background

background.js runs here. Has full Chrome API access and can call the Gmail API — but can be shut down by Chrome when idle to save battery.

3
The web dashboard (coldbase.live)

The React app. Completely separate from the extension — it can't call chrome.runtime at all. It can only talk to the extension through a special relay trick.

Trick 1: Use storage as a message bus

When you send an email, content.js needs to tell background.js. The obvious approach is chrome.runtime.sendMessage() — like a direct phone call. But in MV3, if the background service worker is asleep, the phone rings with nobody home.

Reach's solution: use chrome.storage like a bulletin board instead of a phone call.

CODE 

// content.js — after detecting a send
chrome.storage.local.set({
  coldbase_pending_scan: {
    emailSubject: subject,
    emailRecipients: recipients,
  }
});
PLAIN ENGLISH

After detecting you hit Send, content.js pins a note to the bulletin board.

The note is labeled "coldbase_pending_scan" — background.js watches for exactly this label.

The note contains the email subject and recipients as context.

Chrome wakes up background.js to deliver this storage change — the timing problem is solved.

CODE 

// background.js — always listening
chrome.storage.onChanged.addListener((changes, area) => {
  if (area === 'local' && 'coldbase_pending_scan' in changes) {
    const pendingScan = changes.coldbase_pending_scan?.newValue;
    if (!pendingScan) return;
    chrome.storage.local.remove('coldbase_pending_scan');
    trackLatestSent(false, pendingScan);
  }
});
PLAIN ENGLISH

background.js listens for any change to Chrome's local storage.

If the change is in local storage AND contains our specific bulletin-board note...

Read the note's contents (the email data).

If the note is empty (a deletion event), ignore it.

Remove the note from the board — so we don't process it twice.

Start tracking the sent email using the Gmail API.

💡
Key Insight: Storage as a message bus

Using storage instead of direct messaging is a pattern that solves race conditions caused by background scripts being asleep. If you're ever building an extension and messages aren't arriving, this pattern is worth knowing.

Trick 2: The window message relay

The web dashboard (React) can't call chrome.runtime directly — it's a regular website. But it needs to tell the extension to do things (like trigger a rescan). The solution is a middleman script called dashboard-sync.js.

Think of it like a shipping dock: the React app drops packages through a chute (window.postMessage), the dock worker (dashboard-sync.js) picks them up, re-labels them for the extension (chrome.runtime.sendMessage), and delivers them. The extension sends the response back the same way, in reverse.

Extension permissions — the ID card

Chrome won't let an extension do anything sensitive without declaring it upfront in manifest.json. Think of it like a security clearance form: every capability must be listed. Users see this list when they install the extension.

identity Use Google OAuth to log in users without storing passwords
storage Save data between sessions — used as the message bus between content.js and background.js
alarms Schedule the reply-check to run every 30 minutes, even when Gmail isn't open
scripting Inject content scripts into Gmail tabs — how the sidebar widget gets injected
activeTab Access the currently open tab — only when the user explicitly clicks the extension icon

Adding a new capability to the extension? It almost certainly needs a new permission. Requesting too many permissions is the #1 reason Chrome Web Store rejects extensions — and why users get suspicious. Only ask for what you actually need.

The Clever Tricks

Fuzzy matching, invisible tracking pixels, quota gates, and the MV3 service worker alarm pattern

Trick 1: Fuzzy keyword matching

The classifier doesn't just look for exact words — it uses edit distance to handle typos and spelling variations. So "internships" and "internhsips" both match the "intern" keyword group.

CODE 

function tokenMatchesVariant(token, variant) {
  if (token === variant) return true;
  const limit = variant.length >= 9 ? 2
    : variant.length >= 5 ? 1
    : 0;
  return editDistanceWithinLimit(token, variant, limit);
}
PLAIN ENGLISH

Check if a word from the email matches a keyword variant.

If it's an exact match, immediately return true — no need to do math.

Set a tolerance for typos based on the variant's length:

Long words (9+ chars) allow 2 typos. Medium words (5-8 chars) allow 1. Short words allow none.

Check if the word is close enough using the edit distance algorithm.

The classifier groups keywords by concept — not individual words. So "intern," "interns," "internship," and "interning" all count as ONE match. This prevents an email stuffed with synonyms from scoring artificially high.

Trick 2: Company name extraction via Clearbit

Once Reach knows it's a cold email, it needs the company name. It tries four strategies in order — from fastest to most expensive — and stops as soon as one works. Like a detective working through a list of leads.

1
[Company Name] in brackets

Subject line pattern: "[Anthropic] Software Engineer Role" — extracts instantly, no API call needed.

2
Company — prefix

Subject line pattern: "OpenAI — Interested in joining?" — fast regex match.

3
— Company suffix

Subject line pattern: "Software Engineering — Google" — fast regex match in reverse.

4
Clearbit API lookup

Scans the email body for capitalized word sequences, scores them by frequency, then validates the top candidates against Clearbit. Slowest but catches any company name.

Limitation to know

Many email clients (especially Gmail's web interface) now proxy images through their own servers to block tracking. An "open" event may mean Gmail pre-loaded the images — not that a human actually read the email. Open rate data should be treated as a signal, not ground truth.

Trick 4: The quota gate (free vs pro)

Reach has a freemium model — free users get a limited number of email lookups per month, pro users get more. This is enforced by a middleware function called checkQuota that sits in front of the expensive API endpoints.

CODE 

// server/app.js — rate limiters stack up
app.use('/api/find-email',
  expensiveRateLimit,
  checkQuota
);
app.use('/api/suggest-domains',
  expensiveRateLimit
);
app.use('/api/draft-email',
  expensiveRateLimit
);
PLAIN ENGLISH

For the email-lookup endpoint — the most expensive one — apply TWO guards:

First, rate limiting: max 10 requests per 15 minutes per IP address.

Then, quota check: has this user hit their monthly lookup limit?

Domain suggestions: just rate limited (no quota — less expensive).

AI email drafting: just rate limited (no quota — Gemini is cheap at this scale).

💡
Key Insight: Layered defenses

Notice how Reach stacks two protections: a rate limiter (protects against bursts) and a quota gate (enforces monthly limits). These solve different problems — rate limiting handles abuse in a short window, quotas handle cumulative costs over time. When you ask AI to "add protection" to an endpoint, clarify which type you need.

Trick 5: The reply-check alarm

Reach can't stay connected to Gmail 24/7 — that would drain your battery and violate Chrome's service worker rules. Instead, it uses Chrome's alarm system: a built-in scheduler that fires a named event every N minutes, even when Chrome is in the background.

CODE 

chrome.runtime.onInstalled.addListener(() => {
  chrome.alarms.create('coldbase-reply-check', {
    periodInMinutes: 30
  });
});

chrome.alarms.onAlarm.addListener(async (alarm) => {
  if (alarm.name === 'coldbase-reply-check') {
    const token = await getAuthToken(false);
    await checkReplies(token);
  }
});
PLAIN ENGLISH

When the extension is first installed, set up a repeating alarm...

Give it a name ("coldbase-reply-check") so we can identify it later.

Set it to fire every 30 minutes — automatically, forever.

Whenever any alarm fires, check if it's our reply-check alarm.

If so, get the Gmail access token (silently, without showing a login popup).

Call checkReplies, which scans Gmail for replies to tracked emails.

When Things Break

The most common failure modes, how to debug them, and the full architecture picture

The 5 most common things to go wrong

Every app has a set of failures it's prone to — based on its architecture. Knowing Reach's failure modes turns you from someone who randomly tries things into someone who can diagnose systematically.

1
Email sent but no card appears

Check: Did the classifier flag it as cold outreach? Did the Gmail API return the thread? Is the user authenticated? Is the server running? This is a chain problem — any broken link stops the card.

2
Card stuck on "Sent" when a reply came in

Check: Did the 30-minute alarm fire? Is the access token expired? Did reply-checker.js find the reply in the right Gmail thread? The reply detection uses the Gmail thread ID to match replies to outreach records.

3
Dashboard shows "Failed to load data"

Check: Is the server running? Is the JWT token expired? Is CORS blocking the request? Open the browser DevTools Network tab to see the exact error code.

4
Wrong Gmail account tracked

Reach compares the Gmail account currently open with the Reach account email. If they differ, it shows an "account mismatch" toast and skips tracking. You must be logged into Gmail AS the same account you used to sign up for Reach.

5
"Quota exceeded" on email lookup

Free users have a monthly cap on contact lookups. The server returns a 429 response with the used/limit counts. Solution: upgrade to pro, or wait for the monthly reset (first of the month).

The debugging playbook

When something breaks in Reach, you have three power tools. Use them in order before asking an AI for help — the AI will give a much better answer when you show it real error output.

🔍

Chrome DevTools → Extensions

Open chrome://extensions, find Coldbase, click "Service Worker" to get the background script's console. Every background.js log appears here. This is where to look for Gmail API errors and storage events.

🌐

Gmail Tab → DevTools Console

Open Gmail, right-click → Inspect → Console. content.js logs appear here — search for "[coldbase]" or "[content]". This is where to see email detection decisions and widget errors.

📡

Dashboard → Network Tab

Open coldbase.live, DevTools → Network. Filter by "/api" to see every request the dashboard makes to the server. Look for red rows (error status codes). Click a row to see the full response body — the error message is usually there.

🎯
Debugging tip: Find the broken link

Every problem in Reach is a broken link in a chain. Start from where the user experiences the symptom and trace backwards. "Card not showing" → was it saved? → was it tracked? → was the email detected? You'll find the broken link before you run out of steps.
